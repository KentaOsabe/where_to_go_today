# 要件ドキュメント

## プロジェクト概要（入力）
全体像（`.kiro/specs/overview.md`）に基づき、まず「お店を登録する機能」を作成する。ユーザは食べログのコレクションを元に手動でお店を登録し、提案に使うため食べログのURLを保存する。登録されたお店には「行った店」と「行っていないが気になる店」が混在する。最低限の情報で素早く登録できることを重視する。

## 要件
### 1. 目的
- 1.1 迷わず素早く店を登録できること
- 1.2 提案機能で利用できる最小限の情報を確実に保存できること

### 2. 機能要件
- 2.1 お店を手動で新規登録できる
- 2.2 登録に必要な必須項目は店名・食べログURL・来店ステータスとする
  - 店名
  - 食べログURL
  - 来店ステータス（行った / 行っていない）
- 2.3 追加情報は任意入力とし、後回しにできる
  - ジャンル
  - エリア
  - 予算帯
  - メモ（例: 行った理由、気になるポイント）
- 2.4 保存後に登録結果が確認できる（一覧または詳細へ遷移）
- 2.5 登録に失敗した場合は理由を明確に表示する

### 3. 入力バリデーション
- 3.1 食べログURLは `tabelog.com` ドメインのURLのみ受け付ける
- 3.2 必須項目が未入力の場合は保存できない
- 3.3 同一URLが既に登録済みの場合は重複登録を防止し、既存データの参照を案内する

### 4. ユーザー体験
- 4.1 迷わない入力フロー（1画面で完結、もしくは最小ステップ）
- 4.2 既定値の設定（来店ステータスは「行っていない」を初期値）
- 4.3 キーボード操作中心でも登録完了できる

### 5. データ要件（論理）
- 5.1 店データに以下を保持する
  - 店名（必須）
  - 食べログURL（必須・一意）
  - 来店ステータス（必須）
  - ジャンル（任意）
  - エリア（任意）
  - 予算帯（任意）
  - メモ（任意）
  - 登録日時 / 更新日時

### 6. 非機能要件
- 6.1 登録操作は体感的に速い（保存完了まで待ち時間が長くならない）
- 6.2 入力内容は誤って失われない（バリデーションエラー時に入力値を保持）

### 7. 受け入れ基準
- 7.1 必須項目のみで登録が完了できる
- 7.2 食べログURLが不正な場合に保存されない
- 7.3 重複URLが登録されない
- 7.4 登録直後に保存結果を確認できる

### 対象外
- 外部サービスからの自動インポート
- 登録済みデータの編集・削除（別機能で扱う）
