# 設計レビュー（validate-design）

## 結論
- 要件に対して整合しており、実装に進める内容。
- ルーティングの変更（`/` → `/places` リダイレクト、`/register` 追加）に伴う導線の明確化は実装前に合意しておくと安全。

## 要件→設計の対応
- Req 1（一覧取得と表示）: `GET /api/places`、一覧画面、ローディング/空状態/エラー/再試行の設計で対応済み。
- Req 2（表示情報）: 店名・来店ステータス・食べログURL・任意情報（ジャンル/エリア/予算帯/メモ）を一覧に表示する設計で対応済み。
- Req 3（登録直後の確認）: 登録成功後に `/places` へ遷移し再取得する設計で対応済み。

## ギャップ/懸念
- `/` を一覧へリダイレクトするため、登録導線（`/register` へのリンク/CTA）を一覧画面・空状態に明示する必要がある。
- `page` / `per` の不正値（0/負数/過大）の扱いが設計上明示されていないため、実装時にクランプ/デフォルトのルールを固定する必要がある。

## 仕様上の判断
- ページング（最大50件）と React Router 導入は要件外の追加だが、ギャップ分析の方針と整合。
- メモの省略表示は一覧の可読性を保つためのUI判断であり、詳細画面で全文確認できる前提で問題ない。

## 次のアクション
- 一覧画面に ` /register ` への登録ボタンを追加する（配置は上部）。
- `page` / `per` は 1 以上、`per` は最大 50 のルールで確定する。
