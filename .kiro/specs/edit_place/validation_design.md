# 設計レビュー（validate-design）

## 結論
- 要件に対して整合しており、実装に進められる内容。
- 詳細画面の導線変更（右側アクションの「編集」「削除」）と見出し変更は設計に反映済み。

## 要件→設計の対応
- Req 1（目的）: 編集/削除の導線と結果確認（詳細/一覧遷移）で対応済み。
- Req 2（機能）: 編集対象項目の同等性、編集/削除の完了確認を設計に明記済み。
- Req 3（入力バリデーション）: `tabelog.com` 制約、必須、重複、削除対象なしの 404 を設計に反映済み。
- Req 4（UX）: 既存値の初期表示、削除確認ダイアログ、入力保持の方針を設計に反映済み。
- Req 5（データ要件）: `updated_at` は更新で自動更新、削除は物理削除の前提を設計に明記済み。
- Req 6（非機能）: 失敗時の理由表示と再試行導線を UI/状態管理で対応予定。

## ギャップ/懸念
- 更新時の URL 重複判定は「同一レコードは許可・他レコードは拒否」を明確にする必要があるため、`existing_place_id` の扱い（自分自身を返さない）を実装で担保する。
- 削除失敗時の再試行導線（ボタン再活性化、メッセージ表示の持続）を UI で明確にする必要がある。

## 仕様上の判断
- 編集成功時は詳細、削除成功時は一覧へ遷移する設計は要件の「結果確認」に合致する。
- 詳細画面の見出しを「店舗詳細」に変更しキャプションを省略する判断は UX の簡素化として妥当。

## 次のアクション
- `update` の重複検知で `existing_place_id` を返す条件を実装時に明確化する。
- 削除失敗時の再試行 UI を実装タスクに含める。
